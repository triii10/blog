FVD: A high-performance virtual machine image format for cloud

(2011) - Tang Chunqiang

Link: http://www.usenix.org/events/atc11/tech/final_files/Tang.pdf DOI: Links: Tags: #paper

Abstract

Fast Virtual Disk (FVD) is a new virtual machine (VM) image format and the corresponding block device driver developed for QEMU. QEMU does I/O emulation for multiple hypervisors, including KVM, Xen-HVM, and VirtualBox. FVD is a holistic solution for both Cloud and non-Cloud environments. Its feature set includes flexible configurability, storage thin provisioning without a host file system, compact image, internal snapshot, encryption, copy-on-write, copy-on-read, and adaptive prefetching. The last two features enable instant VM creation and instant VM migration, even if the VM image is stored on direct-attached storage. As its name indicates, FVD is fast. Experiments show that the throughput of FVD is 249% higher than that of QCOW2 when using the Post-Mark benchmark to create files.

Notes

  1. Find out what is dirty-block tracking
  2. Due to storage virtualization, a block address is translated multiple times.
    1. When a guest VM issues an I/O to the hypervisor, it does so using a VBA (virtual block address)
    2. The block device driver of QEMU translates the VBA to IBA (image block address), which is basically where the data is in the image file. (image offset)
    3. If the image is stored in a host file system, then the OS translates the IBA to PBA (physical block address)
    4. If VM is stored in a raw partition, then PBA = IBA (no file system translation)
  3. QCOW2 uses lookup tables to translate VBA to IBA

Summary

  • This paper presents a novel disk format called “Fast Virtual Disk” (FVD).
  • The FVD implements multiple new features, like Copy-On-Read (CoR), and adaptive pre-fetching, which although has been in use but with niche applications such as VM migration, etc.
  • Key finding the authors made is that every disk format mixes the function of storage space allocation with dirty block tracking, which is unnecessary - and increases the average disk seek time by generating more I/O by making unnecessary metadata fetches and the authors attempt to address this issue.
  • Due to mismatch of VBA and IBA, there is undesirable seek distance between accessing files’ metadata, and the files’ content block. FVD eliminates this overhead to optimize
  • FVD Format - metadata: (header & bitmap), datafile (space for data disk & space for expanded data disk). Header contains reference to the backing image in NAS. Bitmap keeps track of which data block is in the backing image and which has been copied and is available in the datafile.
  • It maintains a linear mapping of VBA and IBA, and relies on host OS for storage allocation
  • Further optimizations are made to the function of CoR & CoW to reduce resource consumption and disk I/O
